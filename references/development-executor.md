# TAPD 日常开发执行器

用户要求“实现、写代码、继续开发、按任务做”时，在完成 context resolve、intake 和编码前审核后使用本流程。计划只是输入，目标是把当前可完成的代码、测试和验证真正落地。

## 执行顺序

1. 读取项目级 `AGENTS.md`、现有代码结构和测试命令，不假设技术栈。
2. 确认 `references/pre-dev-review.md` 已执行或用户明确跳过；若存在已确认但未处理的高影响问题，先进入 TAPD 评论 dry-run 或等待用户决定，不直接编码。
3. 确认执行焦点：
   - Task 入口：默认以该 Task 为焦点，同时使用父 Story 约束校验边界。
   - Story 入口：优先使用用户指定 Task；否则从 Story tasks 中选择依赖最前、未完成且与当前仓库相关的一项。
   - 没有 tasks：按 Story must 约束形成最小可验收实现步骤，不擅自创建远端 Task。
4. 检查 `git status`。进入开发后存在普通 dirty change 是正常状态；不得因为已有用户修改而覆盖、回滚或自动 stash。
5. 定位代码落点，实施当前焦点所需的完整最小改动。遵循仓库既有模式，避免无关重构。
6. 按风险补测试并运行仓库已有校验。无法运行时说明具体原因，不把未运行描述为通过。
7. 对照 Story must、当前 Task 和原型逐项复核。
8. 输出：
   - 已完成内容
   - 修改文件
   - 验证结果
   - 尚未完成或待联调项
   - 下一可执行步骤

## “继续开发”

1. 没有新工作项输入时，用 `current --format json` 恢复当前分支工作项。
2. 读取当前 `git diff`、测试结果和会话上下文，判断上次停点。
3. context 第一版不维护自动 progress，不得根据 `local_phase=initialized` 推断尚未开始。
4. 若无法可靠判断下一步，展示已有改动和候选步骤后询问；不要重做已完成代码。

## TAPD 写入

- 实施代码本身不代表授权更新 TAPD 状态。
- 需要创建/更新 Task、评论、工时或排期时，分别走对应 workflow 的 dry-run 与确认。
- 不自动生成 MR/提测说明，不自动提交 `.tapd` 文件。
