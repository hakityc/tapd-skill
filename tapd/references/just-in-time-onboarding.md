# 按需配置与无感引导

当用户提出任何 TAPD/Flow 业务意图时先读本文件。目标是让用户表达业务目标，而不是学习工具命令；`setup`、`doctor`、`init` 是 Agent 的实现细节。

## 自动执行的检查

按以下顺序只读检查，并在成功后继续原请求：

1. 当前目录是否为 Git Repo；产品文档、原型和提交版本是否可作为规格输入。
2. `tapd-context doctor` 的本地状态：项目配置、团队策略、Flow Manifest、分支上下文。
3. MCP 是否在线、可认证，以及当前 workflow 所需的只读/写入工具是否存在。
4. 当前 profile、workspace、写动作权限是否能够从团队策略或已知上下文确定。

不得为“诊断”创建远端工作项、修改 TAPD 状态、发送通知、提交 Git，或把 Token 写入项目文件。

## 只询问缺失的一步

将技术缺口翻译成业务语言，且每轮最多要求一个最小动作：

| 缺失条件 | 对用户的表达 | 后台动作 |
|---|---|---|
| 未连接 TAPD | “需要首次连接你的 TAPD 账号，连接后我会继续创建需求。” | 按 `mcp-bootstrap.md` 配置、reload、重探测并恢复原请求 |
| 无创建/更新权限 | “你的 TAPD 账号目前没有在该项目创建需求的权限。” | 停止远端写入，保留可复制草稿 |
| 产品规格尚未建立 | “我已识别到产品文档和原型；是否以当前已提交版本建立需求草稿？” | 确认后运行 `tapd-context spec init` 与 `spec validate` |
| 开发仓库未初始化 | “我需要确认以哪个基线分支创建需求分支。” | 展示候选；确认后运行 `tapd-context init` 并恢复原请求 |
| profile 或 workspace 不明确 | “这次以产品/前端/后端/测试哪个角色处理？目标 TAPD 项目是哪一个？” | 仅保存低敏个人偏好；重新执行门禁 |

不要把上表中的内部命令、文件名或错误码作为用户必须理解的前置条件；除非用户要求技术细节。

## 确认边界

以下操作可以自动执行：读取仓库和已提交规格、生成草稿、执行诊断、只读查询、重新探测、恢复原请求。

以下操作需要明确确认：

1. 首次连接或修改个人 MCP 配置。
2. 固化当前产品 commit 为规格 Manifest，或以某个基线创建开发分支。
3. 所有远端写入：创建/更新 Requirement、Task、TestCase、Comment、工时、状态。

确认文案应说明结果而非命令。例如说“是否以当前版本建立需求草稿”，不说“是否执行 spec init”。对外写入仍遵守 dry-run、二次确认和写后回读。

## 恢复与降级

配置、reload 或权限修复成功后，必须自动重复诊断并继续用户的原始业务请求。仍无法恢复时，输出已完成的本地草稿、阻塞原因和唯一可执行的下一步；不得假装 TAPD 已写入，也不得要求用户从头复述需求。

没有 MCP 时允许分析已提交的产品文档、原型和本地 Manifest，并生成需求/测试草稿；不得绕过 MCP 直接调用 TAPD OpenAPI。
