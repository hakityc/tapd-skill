# TAPD 编码前产品/原型审核

在用户要求“实现、写代码、按任务做、继续开发”前，或用户明确要求“先审一下产品文档/原型/差异点/返工风险”时使用本流程。目标是把真正会影响开发、联调、验收或返工的问题提前暴露出来，并在用户确认后同步到 TAPD 评论。

## 前置

- 必须先完成 `references/intake-gate.md`，拿到 Story description、原型提取、Story ↔ Tasks 对照和代码落点初筛。
- 按 `references/mcp-capability-matrix.md` 检查能力；评论写入需要 `create_comments`，缺失时只输出评论草案。
- 所有 TAPD 写入遵守 `references/safety-policy.md`：先 dry-run，未经确认不写。
- Task 入口时，审核对象仍以父 Story 为边界；先执行 `references/spec-authority.md`，正式准入时以 reviewed_ref 产品规格为需求真相，当前 Task 只作为执行焦点。

## 审核范围

只找会实际影响开发推进或验收结果的问题：

- **需求 ↔ 原型差异**：字段、按钮、状态、权限、流程入口、默认值、异常态在文档和原型中不一致。
- **验收口径缺失或冲突**：must 条款无法映射到可验证行为，或多个条款给出相反规则。
- **接口/数据阻塞**：页面需要的字段、状态枚举、权限点、保存/查询接口、事件回调没有来源或命名不一致。
- **任务拆解风险**：已有 tasks 漏掉必须开发的前后端协作点，或 task 描述与 Story/原型相冲突。
- **权限/状态矩阵风险**：不同角色、状态、开关组合会改变可见性或可操作性，但需求没有给出清晰规则。
- **迁移/兼容风险**：涉及历史数据、已有配置、旧入口、灰度开关、回滚策略，但需求未说明处理方式。

明确不做：

- 不挑错别字、文案润色、命名偏好、视觉细节偏差，除非它们直接影响字段含义、接口契约或验收。
- 不把“可以更好”当问题；必须说明不处理会造成的具体开发、联调或验收影响。
- 不擅自扩需求；只能提出确认问题或建议补充验收口径。

## 严重度门槛

每个候选问题必须满足至少一个条件，否则丢弃：

- 会导致实现路径不同，已经写完后可能大改。
- 会阻塞前后端联调、接口设计或数据模型。
- 会导致测试无法判断通过/失败。
- 会影响权限、状态、数据安全、历史兼容或线上回滚。
- 会让当前 tasks 明显漏掉必须交付的工作。

输出时按影响排序，最多列 8 条；宁可少报，不要铺满低价值问题。

## 审核步骤

1. 对照 intake 结果建立四张表：Story must、原型交互、已有 tasks、代码落点/API/类型。
2. 找出候选差异和缺口，并为每条写清：
   - 来源：Story 条款 / 原型元素 / Task / 代码现状。
   - 影响：会阻塞什么、导致什么返工或验收歧义。
   - 需要谁确认：产品 / 后端 / 测试 / 研发。
   - 建议确认问题：一句可直接贴进 TAPD 的问题。
3. 过滤低价值项，只保留满足严重度门槛的问题。
4. 向用户输出“待确认问题清单”，等待用户确认哪些是真问题。
5. 用户确认后，生成 TAPD 评论 dry-run；用户授权写入且有 `create_comments` 时写到 Story 评论。
6. 写入后按能力回读；无法回读时说明“写入成功，但无法评论列表回读”。

## 用户确认规则

- 未经用户明确确认，不把候选问题写入 TAPD。
- 用户可用编号确认，例如“1、3 是问题，2 先忽略”；只同步被确认的编号。
- 用户要求“直接同步全部”时，仍先展示简短 dry-run，再按 `safety-policy.md` 执行。
- 如果用户判断“都不是问题”，不得写评论；继续计划或编码时把审核结果记为无确认风险。

## 输出模板（候选问题）

```text
**编码前审核**

结论：发现 N 个可能导致返工/阻塞的问题，待你确认；低影响项已过滤。

1. 【严重度：高/中】<一句话问题>
   来源：<Story/原型/Task/代码落点>
   影响：<具体返工、联调阻塞或验收歧义>
   建议确认：<可直接问产品/后端/测试的问题>

待确认：请回复要同步到 TAPD 的编号，例如“1、3 同步，2 忽略”。
```

## TAPD 评论格式

确认后写入 Story 评论，标题固定使用：

```text
【编码前审核】产品文档/原型待确认问题
```

推荐 HTML：

```html
<p><b>【编码前审核】</b>产品文档/原型待确认问题</p>
<p><b>背景：</b>编码前对 Story、原型和已有 tasks 做一致性审核；以下仅包含已确认会影响开发、联调或验收的问题。</p>
<p><b>确认问题：</b></p>
<ol>
<li><b>问题：</b>...<br><b>影响：</b>...<br><b>建议确认：</b>...</li>
</ol>
<p><b>关联：</b>Story：完整 TAPD URL</p>
<p><b>收尾：</b>确认结果补齐后，再进入实现计划或编码。</p>
```

纯文本降级：

```text

背景：

确认问题：
1. 问题：...
   影响：...
   建议确认：...

关联：
- Story：<完整 TAPD URL>

收尾：
确认结果补齐后，再进入实现计划或编码。
```

## 失败与降级

- 缺少 `create_comments`：不得写 TAPD，输出可复制评论正文，并建议按 `references/mcp-bootstrap.md` 修复 MCP。
- 缺少原型图片读取能力：审核可以继续，但必须在候选问题中标明“原型图片未读取，无法判断视觉/交互细节”这类 gap；不能假装已看过。
- 需求信息不足：只输出已能判断的高影响问题，不用猜测补齐。
