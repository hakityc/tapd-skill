# Flow：团队研发交付工作流

## 定位

Flow 是以研发工作项为中心，连接产品规格、评审、前后端开发、测试和发布验证的 AI 团队协作工作流。

TAPD 是第一个 Work Item Provider，不是产品边界。未来可以通过 Provider Adapter 接入飞书、Jira 或其他系统，而产品 Git Repo、代码仓库和测试证据保持不变。

## 事实源

- 产品 Git Repo：完整规格、原型、验收点和评审决定。
- Work Item Provider：需求摘要、任务、评论、负责人、进度和风险。
- 代码仓库：实现、分支、提交和自动化测试。
- 测试系统：Case、环境、结果和证据。

## 产品承诺

团队成员从一个工作项可以追溯到：

- 它基于哪个产品规格版本。
- 评审确认了什么、还有什么风险。
- 前端、后端和质量分别交付什么。
- 哪些代码与测试对应哪些验收点。
- 当前是否具备发布条件。

## 核心原则

1. 规格单向发布，避免 Git 与协作平台自由双向编辑。
2. 所有正式执行都固定到精确规格版本。
3. AI 可以分析和生成草案，关键 Gate 必须由人确认。
4. 核心工作流使用通用 Requirement、Task、Defect、TestCase 和 Evidence；平台字段留在 Adapter。
5. Provider 缺少能力时显式降级，不绕过权限或伪装成功。
6. 度量流程瓶颈，不做个人绩效排名。

## 北极星指标

从产品评审通过到达到可发布状态的中位周期。

护栏指标：返工率、生产逃逸缺陷、写入撤销率、未验证验收点数量。
