# TAPD Skill 团队使用手册

> 内部试用版。推荐使用 `/tapd` 作为入口；也可以直接用自然语言说明目标。不同 Agent 对 `/TAPD` 的大小写兼容不完全一致，因此请优先使用小写 `/tapd`。

## 30 秒开始

首次安装：

```bash
npx skills add hakityc/tapd-skill --skill tapd --global --yes
```

之后，进入对应的产品或代码仓库，直接输入业务目标即可。例如：

```text
/tapd 根据当前产品文档和原型创建需求
```

AI 会先检查仓库、文档、原型、TAPD 连接和权限。首次使用时如需连接 TAPD，会只提示当前缺少的一步；请通过安全授权方式完成，不要把 Token 发到聊天、文档或仓库。

## 所有人都应知道的规则

1. 产品文档和原型先提交到产品 Git 仓库，再创建或更新需求。
2. AI 可以自动分析、生成草稿和检查环境；创建或更新 TAPD 内容前一定会展示预览，需人工确认。
3. 产品 Git 仓库保存完整规格；TAPD 保存协作摘要、任务、评审结论和进度。
4. 评审通过只由人确认；AI 不会自行宣布需求已通过或已经发布。
5. 文档或原型修改后，先提交新版本，再让 AI 检查影响并重新发布/增量评审。

## 产品经理：标准试用主线

按下面五步走即可完成一次真实需求的试用。

| 阶段 | 直接输入 | AI 会产出 | 你需要做什么 |
|---|---|---|---|
| 1. 建立需求草稿 | `/tapd 根据当前产品文档和原型创建需求` | 目标、范围、非范围、验收标准草案 | 检查并确认范围与验收标准 |
| 2. 发布需求 | `/tapd 从当前产品仓库发布需求` | TAPD 需求预览、版本信息、验收标准 | 确认预览后创建/更新 |
| 3. 准备评审 | `/tapd 准备产品评审` | 评审包、差异、依赖、待决问题、任务草案 | 用它组织评审并记录结论 |
| 4. 记录结论 | `/tapd 记录评审结论：approved` | 将写入的规格版本与 TAPD 内容预览 | 确认评审确实通过后再写入 |
| 5. 需求变更 | `/tapd 检查这次产品修改是否需要重新评审` | 版本差异、受影响范围、增量评审建议 | 确认需重新评审的范围 |

### 第一次创建需求

在产品 Git 仓库中输入：

```text
/tapd 根据当前产品文档和原型创建需求。
产品文档：<产品文档路径>
原型：<原型路径>
目标 TAPD 项目：<项目编号或项目名称>
先给我看范围和验收标准草案，确认后再创建。
```

如果仓库中只有一份明确的产品文档和原型，可以省略路径。AI 会在缺少信息时只追问一项。

### 发布到 TAPD

```text
/tapd 从当前产品仓库发布需求。先给我看预览，确认后创建或更新。
```

预期结果：AI 检查规格版本与路径，查找是否已存在同一份需求；存在时更新同一条，避免重复创建。

### 准备和记录产品评审

```text
/tapd 准备产品评审，列出高影响差异、依赖、待决问题和前端、后端、测试任务草案。
```

评审结束后：

```text
/tapd 记录评审结论：approved。
先展示将要更新的内容，确认后再写入。
```

若结论不是通过，可把 `approved` 替换为 `changes-required`，并补充已确认的修改项。

### 产品修改后的标准动作

先修改并提交产品文档/原型，再输入：

```text
/tapd 检查当前产品规格与已评审版本的差异；列出受影响的验收标准、任务和测试范围，并建议是否增量评审。
```

确认后再发布新版本：

```text
/tapd 发布当前产品文档的新版本。先看预览，确认后更新已有需求。
```

## 常用场景速查

| 你想做什么 | 推荐输入 |
|---|---|
| 从文档和原型生成需求草稿 | `/tapd 根据当前产品文档和原型创建需求` |
| 发布或同步产品需求 | `/tapd 从当前产品仓库发布需求` |
| 再次发布但不想重复创建 | `/tapd 再发布一次当前需求，确认是否需要更新` |
| 准备产品评审材料 | `/tapd 准备产品评审` |
| 记录评审通过 | `/tapd 记录评审结论：approved` |
| 记录需要修改 | `/tapd 记录评审结论：changes-required，并列出待修改项` |
| 比对评审后的变更 | `/tapd 检查当前规格和已评审版本的差异` |
| 询问当前需求状态 | `/tapd 查看当前产品需求的发布和评审状态` |
| 让 AI 先只出草稿 | `/tapd 只生成需求草稿，不写入 TAPD` |

## 团队完整功能清单

| 角色 | 场景 | 推荐输入 | 当前行为 |
|---|---|---|---|
| 产品 | 规格建模与发布 | `/tapd 根据当前产品文档和原型创建需求` | 生成规格、验收标准和 TAPD 需求预览 |
| 产品 | 产品评审 | `/tapd 准备产品评审` | 生成评审包，人工确认后冻结规格版本 |
| 前端/后端 | 开始需求 | `/tapd 开始做 <Story 或 Task 链接>` | 读取需求和原型，建立本机上下文与分支 |
| 前端/后端 | 编码前审核 | `/tapd 先审一下产品文档和原型差异` | 只报告可能造成返工、联调阻塞或验收失败的问题 |
| 前端/后端 | 拆任务与排期 | `/tapd 合理拆分任务并回填 TAPD` | 生成任务、负责人、估时和排期预览；确认后写回 |
| 前端/后端 | 继续开发 | `/tapd 继续开发` | 从当前分支恢复上下文，继续计划、实现和验证 |
| 测试/质量 | 生成测试用例 | `/tapd 基于当前需求生成测试用例` | 基于需求和验收标准生成 Case 草稿或写入（取决于权限和 MCP 能力） |
| 研发 | 进度同步 | `/tapd 把已确认的问题同步到 TAPD 评论` | 生成评论预览，确认后写入 |
| 研发 | 收尾与工时 | `/tapd 这个需求做完了，帮我收尾并登记 <时长> 工时` | 执行验证、生成摘要和工时预览；不自动流转状态 |
| 任意成员 | 日报/站会 | `/tapd 生成今天的站会简报` | 只读汇总完成、进行中、风险和下一步 |
| 任意成员 | 今日待办 | `/tapd 看看我今天有哪些活要干` | 从当前迭代未完成任务给出优先级排序 |
| 负责人 | 团队盘点 | `/tapd 看下当前迭代团队负载和风险` | 只读汇总 WIP、逾期、阻塞和无人负责事项 |

## 写入前会发生什么

下面操作会先展示预览，再等待确认：

- 创建或更新 TAPD 需求、任务、测试用例、评论、工时。
- 固定一份产品规格版本。
- 创建开发分支。

以下操作可以直接完成，不会改动 TAPD：读取需求、分析文档和原型、生成草稿、生成评审包、检查差异、日报、今日待办和团队风险盘点。

## 遇到问题时怎么说

不要自己查命令。直接把现象告诉 AI：

```text
/tapd 我想创建需求，但当前无法连接 TAPD。请检查并告诉我只需要完成哪一步。
```

或：

```text
/tapd 这次发布没有成功。请保留草稿，检查原因，并告诉我下一步怎么处理。
```

AI 应保留已生成的草稿，说明阻塞原因，并在连接或权限恢复后继续原始请求；不会因为失败重复创建需求。

## 当前试用边界

- 当前协作平台是 TAPD；飞书等平台尚未接入。
- 不自动发送通知、不自动流转需求状态、不自动提交 Git。
- 没有权限或缺少 MCP 能力时，保留草稿并停止写入，不会绕过平台直接写数据。
- 首次授权 TAPD 仍需本人操作；AI 不保存任何 Token。
